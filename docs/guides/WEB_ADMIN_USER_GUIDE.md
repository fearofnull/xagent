## 简介

Web 管理界面是飞书 AI Bot 的可视化配置管理工具，提供比命令行和飞书消息命令更友好的操作体验。通过 Web 界面，你可以：

- 📋 查看所有会话的配置列表
- ✏️ 编辑会话配置
- 🔄 重置配置到默认值
- 🔍 搜索和筛选配置
- 💾 导出和导入配置数据
- 🌐 查看全局默认配置

### 访问地址

- **开发环境**: http://localhost:5173 (前端开发服务器)
- **生产环境**: http://your-server:5000 (根据实际部署配置)

## 登录和认证

### 首次登录

1. **打开浏览器**，访问 Web 管理界面地址
2. **输入管理员密码**
   - 密码在服务器的 `.env` 文件中配置（`WEB_ADMIN_PASSWORD`）
   - 如果不知道密码，请联系系统管理员
3. **点击"登录"按钮**
4. 登录成功后会自动跳转到配置列表页面

### 认证机制

Web 管理界面使用 **JWT (JSON Web Token)** 令牌进行身份验证：

- **令牌有效期**: 默认 2 小时
- **自动存储**: 令牌会自动保存在浏览器的 localStorage 中
- **自动携带**: 每次 API 请求会自动携带令牌
- **过期处理**: 令牌过期后需要重新登录

### 登出

当你完成配置管理工作后，建议主动登出：

1. 点击页面右上角的 **"登出"** 按钮
2. 系统会清除本地存储的令牌
3. 自动跳转回登录页面

**安全提示**：
- 在公共电脑上使用后，务必记得登出
- 不要与他人分享管理员密码
- 定期更换管理员密码（建议 3-6 个月）

### 令牌过期

如果在使用过程中看到"令牌已过期，请重新登录"的提示：

1. 这是正常的安全机制（默认 2 小时后过期）
2. 点击"确定"按钮
3. 系统会自动跳转到登录页面
4. 重新输入密码登录即可

**延长令牌有效期**（需要管理员权限）：
在服务器的 `.env` 文件中设置：
```bash
JWT_EXPIRATION=14400  # 4 小时（单位：秒）
```

## 配置管理操作

### 查看配置列表

登录成功后，你会看到所有会话的配置列表：

#### 列表信息

| 列名 | 说明 | 示例 |
|------|------|------|
| 会话 ID | 用户 ID（私聊）或群组 ID（群聊） | `ou_xxx` 或 `oc_xxx` |
| 会话类型 | `user`（私聊）或 `group`（群聊） | user |
| 更新时间 | 配置最后更新的时间 | 2024-01-01 12:00:00 |

#### 工具栏功能

- 🔍 **搜索框**: 输入会话 ID 进行搜索
- 🏷️ **类型筛选**: 选择"全部"、"私聊"或"群聊"
- 🔄 **刷新按钮**: 获取最新的配置数据
- 📥 **导出配置**: 导出所有配置为 JSON 文件
- 📤 **导入配置**: 从 JSON 文件导入配置

#### 搜索和筛选

**按会话 ID 搜索**：
1. 在搜索框中输入会话 ID（支持部分匹配）
2. 列表会实时过滤显示匹配的结果
3. 清空搜索框可恢复显示所有配置

**按会话类型筛选**：
1. 点击"会话类型"下拉框
2. 选择"全部"、"私聊"或"群聊"
3. 列表会显示对应类型的配置

**组合使用**：
可以同时使用搜索和筛选功能，例如：
- 筛选"私聊" + 搜索"ou_123" = 查找特定用户的私聊配置

#### 排序

点击"更新时间"列标题可以切换排序方式：
- **升序**: 最早更新的配置在前
- **降序**: 最新更新的配置在前（默认）

### 查看配置详情

点击配置列表中的任意一行，进入配置详情页面。

#### 配置信息

详情页面分为三个部分：

**1. 配置字段**

| 字段 | 说明 | 示例 |
|------|------|------|
| target_project_dir | CLI 工具的目标项目目录 | `/path/to/project` |
| response_language | AI 回复语言 | `zh-CN` |
| default_provider | 默认 AI 提供商 | `claude` |

| default_cli_provider | CLI 层专用提供商 | `gemini` |

**2. 元数据**

| 字段 | 说明 |
|------|------|
| created_by | 创建者的用户 ID |
| created_at | 配置创建时间 |
| updated_by | 最后更新者的用户 ID |
| updated_at | 最后更新时间 |
| update_count | 配置被更新的次数 |

**3. 有效配置**

显示应用优先级规则后的最终配置值：
- 会话配置 > 全局配置 > 系统默认值
- 标注哪些字段使用了会话配置，哪些使用了默认值

### 编辑配置

在配置详情页面，点击 **"编辑"** 按钮进入编辑模式。

#### 编辑步骤

1. **修改字段值**
   - **target_project_dir**: 输入项目目录的绝对路径
   - **response_language**: 输入语言代码（如 `zh-CN`、`en-US`）
   - **default_provider**: 从下拉框选择 `claude`、`gemini` 或 `openai`

   - **default_cli_provider**: 从下拉框选择 `claude`、`gemini`、`openai` 或留空

2. **保存更改**
   - 点击 **"保存"** 按钮
   - 系统会验证配置值的有效性
   - 验证通过后保存配置
   - 显示成功提示并刷新页面

3. **取消编辑**
   - 点击 **"取消"** 按钮
   - 放弃所有未保存的更改
   - 返回查看模式

#### 验证规则

系统会自动验证配置值：

| 字段 | 验证规则 |
|------|----------|
| default_provider | 必须是 `claude`、`gemini` 或 `openai` |

| default_cli_provider | 可以为空，或是 `claude`、`gemini`、`openai` |
| target_project_dir | 检查路径格式（警告但不阻止保存） |

**验证失败时**：
- 字段下方会显示红色错误提示
- 字段边框会变红高亮
- 无法提交表单，需要修正错误

**目录不存在警告**：
- 如果输入的目录路径不存在，会显示黄色警告
- 但仍然允许保存（因为目录可能在其他服务器上）

#### 配置字段说明

**target_project_dir**（目标项目目录）：
- 用于 CLI 层工具（如 `@code` 命令）
- 必须是绝对路径
- 示例：`E:\Projects\my-project` 或 `/home/user/projects/my-project`

**response_language**（回复语言）：
- AI 回复时使用的语言
- 常用值：`zh-CN`（简体中文）、`en-US`（英语）
- 也可以用自然语言描述，如"中文"、"English"

**default_provider**（默认 AI 提供商）：
- 不指定命令前缀时使用的 AI
- `claude`: Claude AI（推荐用于复杂推理）
- `gemini`: Google Gemini（推荐用于多模态任务）
- `openai`: OpenAI GPT（推荐用于通用对话）

- 不指定命令前缀时使用的执行层
- `api`: API 层（快速对话，秒级响应）
- `cli`: CLI 层（代码操作，分钟级响应）

**default_cli_provider**（CLI 层专用提供商）：
- 使用 CLI 层时优先使用的 AI
- 可以为空（使用 default_provider）
- 用于区分 API 和 CLI 使用不同的 AI

### 重置配置

如果想清除某个会话的自定义配置，恢复使用全局默认值：

#### 重置步骤

1. 在配置详情页面，点击 **"重置配置"** 按钮
2. 系统会显示确认对话框：
   ```
   确定要重置此配置吗？
   重置后该会话将使用全局默认配置。
   此操作不可撤销。
   ```
3. 点击 **"确定"** 执行重置，或点击 **"取消"** 放弃操作
4. 重置成功后会显示提示并返回配置列表页面

#### 重置效果

- 删除该会话的所有自定义配置
- 会话将使用全局默认配置
- 配置文件中不再保存该会话的记录
- 操作不可撤销（但可以重新设置配置）

**注意**：重置配置不会影响该会话的对话历史，只是清除配置设置。

### 查看全局配置

点击导航栏的 **"全局配置"** 菜单，查看系统的全局默认配置。

#### 全局配置说明

- **来源**: 来自服务器的 `.env` 文件
- **作用**: 作为所有会话配置的默认值
- **优先级**: 会话配置 > 全局配置 > 系统默认值
- **只读**: 全局配置不能通过 Web 界面修改

#### 修改全局配置

如需修改全局配置：

1. 编辑服务器上的 `.env` 文件
2. 修改以下环境变量：
   ```bash
   TARGET_PROJECT_DIR=/path/to/project
   RESPONSE_LANGUAGE=zh-CN
   DEFAULT_PROVIDER=claude
   DEFAULT_CLI_PROVIDER=gemini
   ```
3. 重启 Web 服务器使配置生效

详细配置说明请参考 [配置文档](CONFIGURATION.md)。

## 导出导入操作

### 导出配置

导出功能用于备份配置数据或迁移到其他环境。

#### 导出步骤

1. 在配置列表页面，点击 **"导出配置"** 按钮
2. 系统会生成包含所有配置的 JSON 文件
3. 浏览器会自动下载该文件
4. 文件名格式：`configs_export_YYYYMMDD_HHMMSS.json`

#### 导出内容

导出的 JSON 文件包含：
- 所有会话的配置数据
- 配置字段（target_project_dir、response_language 等）
- 元数据（创建时间、更新时间、更新次数等）

#### 导出文件示例

```json
{
  "configs": [
    {
      "session_id": "ou_xxx",
      "session_type": "user",
      "config": {
        "target_project_dir": "/path/to/project",
        "response_language": "zh-CN",
        "default_provider": "claude",

        "default_cli_provider": null
      },
      "metadata": {
        "created_by": "ou_xxx",
        "created_at": "2024-01-01T00:00:00Z",
        "updated_by": "ou_xxx",
        "updated_at": "2024-01-01T12:00:00Z",
        "update_count": 5
      }
    }
  ],
  "export_time": "2024-01-01T12:00:00Z",
  "total_count": 1
}
```

#### 导出用途

- **备份**: 定期备份配置数据，防止意外丢失
- **迁移**: 将配置从测试环境迁移到生产环境
- **版本控制**: 将配置文件纳入版本控制系统
- **审计**: 保留配置变更的历史记录
- **恢复**: 在配置出错时快速恢复到之前的状态

### 导入配置

导入功能用于从备份文件恢复配置或从其他环境迁移配置。

#### 导入步骤

1. 在配置列表页面，点击 **"导入配置"** 按钮
2. 在文件选择对话框中，选择之前导出的 JSON 文件
3. 系统会验证文件格式和内容
4. 验证通过后开始导入
5. 显示导入结果（成功导入的配置数量）

#### 导入前备份

**重要**：系统会在导入前自动备份现有配置！

- 备份文件保存在 `data/` 目录
- 文件名格式：`session_configs_backup_YYYYMMDD_HHMMSS.json`
- 如果导入出错，可以手动恢复备份文件

#### 导入规则

- **覆盖规则**: 如果导入的配置与现有配置的 session_id 相同，会覆盖现有配置
- **新增规则**: 如果导入的配置不存在，会创建新配置
- **元数据**: 导入时会保留原有的元数据（创建时间、更新次数等）

#### 验证规则

系统会验证导入文件：

| 验证项 | 说明 |
|--------|------|
| JSON 格式 | 必须是有效的 JSON 格式 |
| 必需字段 | 必须包含 `configs` 数组 |
| 配置结构 | 每个配置必须包含 `session_id`、`session_type`、`config` |
| 配置值 | provider、layer 等字段必须是有效值 |

**验证失败时**：
- 显示错误提示，说明具体的错误原因
- 不会导入任何配置
- 现有配置不受影响

#### 导入注意事项

1. **检查文件**: 导入前确认文件是正确的导出文件
2. **备份确认**: 导入前确认系统已自动创建备份
3. **测试环境**: 建议先在测试环境验证导入效果
4. **逐步导入**: 对于大量配置，可以分批导入
5. **验证结果**: 导入后检查配置是否正确

### 导出导入最佳实践

#### 定期备份

建议定期导出配置作为备份：
- **频率**: 每周或每月一次
- **存储**: 保存到安全的位置（如云存储、版本控制系统）
- **命名**: 使用有意义的文件名，包含日期和环境信息
  - 示例：`configs_production_20240101.json`

#### 环境迁移

从测试环境迁移到生产环境：

1. **导出测试环境配置**
   ```
   测试环境 → 导出配置 → configs_test.json
   ```

2. **审查配置内容**
   - 检查目录路径是否需要调整
   - 确认 AI 提供商配置是否正确
   - 验证语言设置

3. **导入到生产环境**
   ```
   configs_test.json → 导入配置 → 生产环境
   ```

4. **验证导入结果**
   - 检查配置列表
   - 测试几个会话的配置是否生效

#### 版本控制

将配置纳入版本控制：

```bash
# 导出配置
# 在 Web 界面点击"导出配置"

# 将导出文件添加到 Git
git add configs/configs_export_20240101.json
git commit -m "Backup configs: 2024-01-01"
git push
```

#### 灾难恢复

如果配置文件损坏或误删除：

1. **停止服务器**
   ```bash
   # 停止 Web 服务器
   ```

2. **恢复备份文件**
   ```bash
   # 从备份恢复
   cp data/session_configs_backup_20240101.json data/session_configs.json
   
   # 或从导出文件恢复（需要通过 Web 界面导入）
   ```

3. **重启服务器**
   ```bash
   python -m xagent.web_admin.server
   ```

4. **验证恢复结果**
   - 登录 Web 界面
   - 检查配置列表
   - 测试配置是否正常工作

## 常见问题解答

### 登录和认证相关

#### Q1: 忘记管理员密码怎么办？

**A**: 管理员密码存储在服务器的 `.env` 文件中：

1. 登录到服务器
2. 编辑 `.env` 文件
3. 修改 `WEB_ADMIN_PASSWORD` 的值
4. 重启 Web 服务器
5. 使用新密码登录

示例：
```bash
# 编辑 .env 文件
nano .env

# 修改密码
WEB_ADMIN_PASSWORD=new_secure_password

# 重启服务器
python -m xagent.web_admin.server
```

#### Q2: 为什么登录后一段时间就需要重新登录？

**A**: 这是正常的安全机制。JWT 令牌默认有效期为 2 小时，过期后需要重新登录。

**延长令牌有效期**：
在 `.env` 文件中设置：
```bash
JWT_EXPIRATION=14400  # 4 小时（单位：秒）
```

#### Q3: 可以有多个管理员账号吗？

**A**: 当前版本只支持单个管理员密码。所有管理员共享同一个密码。

如需多用户支持，请提交 Feature Request 或自行扩展认证模块。

#### Q4: 在公共电脑上使用后如何确保安全？

**A**: 使用完毕后务必：
1. 点击右上角的"登出"按钮
2. 关闭浏览器窗口
3. 清除浏览器缓存（可选）

#### Q5: 令牌存储在哪里？安全吗？

**A**: 令牌存储在浏览器的 localStorage 中。

**安全性**：
- localStorage 只能被同源的 JavaScript 访问
- 使用 HTTPS 可以防止令牌在传输过程中被窃取
- 令牌有过期时间，降低被盗用的风险
- 建议在生产环境使用 HTTPS

### 配置管理相关

#### Q6: 修改配置后多久生效？

**A**: 立即生效。配置保存后，下一次 AI 请求就会使用新配置。

#### Q7: Web 界面和飞书消息命令修改配置有什么区别？

**A**: 两者操作的是同一个配置文件，效果完全相同。

**选择建议**：
- **Web 界面**: 适合批量管理、查看历史、导出导入
- **飞书命令**: 适合快速修改单个配置，无需离开聊天窗口

#### Q8: 可以同时使用 Web 界面和飞书命令修改配置吗？

**A**: 可以。两者修改会立即同步。但建议：
- 避免同时修改同一个配置（可能导致覆盖）
- 修改后在 Web 界面点击"刷新"查看最新状态

#### Q9: 重置配置后可以恢复吗？

**A**: 重置操作不可撤销，但你可以：
1. 重新设置配置（手动输入）
2. 从导出的备份文件恢复（如果有备份）

**建议**：重置前先导出配置作为备份。

#### Q10: 配置中的目录路径不存在会怎样？

**A**: 系统会显示警告，但仍然允许保存。

**原因**：
- 目录可能在其他服务器上
- 目录可能稍后创建
- 允许灵活配置

**建议**：确保目录路径正确，否则 CLI 层功能可能无法正常工作。

#### Q11: 如何查看配置的修改历史？

**A**: 当前版本不支持查看完整的修改历史，但可以查看：
- **update_count**: 配置被修改的次数
- **updated_at**: 最后修改时间
- **updated_by**: 最后修改者

**建议**：
- 定期导出配置并纳入版本控制（Git）
- 使用有意义的提交消息记录修改原因

#### Q12: 配置字段可以留空吗？

**A**: 可以。留空的字段会使用全局默认值。

**示例**：
- 如果 `default_provider` 留空，会使用全局配置的 `DEFAULT_PROVIDER`
- 如果全局配置也没有，会使用系统默认值（`claude`）

#### Q13: 如何为新用户或新群组创建配置？

**A**: 有两种方式：

**方式 1：通过飞书消息命令**（推荐）
```
/config set default_provider claude
```
用户或群组首次使用配置命令时会自动创建配置。

**方式 2：通过 API**（高级用户）
使用 PUT 请求创建新配置：
```bash
curl -X PUT http://localhost:5000/api/configs/ou_xxx \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"default_provider": "claude"}'
```

**注意**：Web 界面目前不支持直接创建新配置，只能编辑已存在的配置。

### 导出导入相关

#### Q14: 导出的配置文件可以手动编辑吗？

**A**: 可以，但需要注意：
1. 保持 JSON 格式正确
2. 不要修改必需字段的结构
3. 配置值必须符合验证规则
4. 编辑后建议先在测试环境导入验证

**建议**：使用专业的 JSON 编辑器（如 VS Code）编辑，避免格式错误。

#### Q15: 导入配置会覆盖现有配置吗？

**A**: 是的。如果导入的配置与现有配置的 session_id 相同，会覆盖现有配置。

**保护措施**：
- 系统会在导入前自动备份现有配置
- 备份文件保存在 `data/` 目录
- 可以从备份文件恢复

#### Q16: 导入失败了怎么办？

**A**: 导入失败的常见原因和解决方法：

| 错误 | 原因 | 解决方法 |
|------|------|----------|
| JSON 格式错误 | 文件不是有效的 JSON | 使用 JSON 验证工具检查文件 |
| 缺少必需字段 | 文件结构不完整 | 确保包含 `configs` 数组 |
| 配置值无效 | provider 等字段值不正确 | 检查并修正配置值 |
| 文件过大 | 配置数量太多 | 分批导入或增加服务器内存 |

**调试步骤**：
1. 检查浏览器控制台的错误信息
2. 查看服务器日志 `logs/web_admin_error.log`
3. 使用在线 JSON 验证工具验证文件格式
4. 尝试导入一个简单的测试文件

#### Q17: 可以只导入部分配置吗？

**A**: 可以。手动编辑导出文件，只保留需要导入的配置：

```json
{
  "configs": [
    {
      "session_id": "ou_xxx",
      "session_type": "user",
      "config": { ... },
      "metadata": { ... }
    }
  ]
}
```

删除不需要导入的配置项，然后导入编辑后的文件。

#### Q18: 导入配置后如何验证？

**A**: 导入后建议进行以下验证：

1. **检查配置列表**
   - 确认导入的配置数量正确
   - 检查会话 ID 是否正确

2. **查看配置详情**
   - 随机抽查几个配置
   - 确认配置字段值正确

3. **测试功能**
   - 在飞书中测试几个会话
   - 确认配置生效

4. **检查日志**
   - 查看 `logs/web_admin.log`
   - 确认没有错误信息

### 技术和故障排查

#### Q19: Web 界面无法访问怎么办？

**A**: 按以下步骤排查：

1. **检查服务器是否运行**
   ```bash
   # 检查进程
   ps aux | grep web_admin  # Linux/Mac
   tasklist | findstr python  # Windows
   ```

2. **检查端口是否正确**
   - 默认端口：5000
   - 确认浏览器访问的端口与服务器监听的端口一致

3. **检查防火墙**
   ```bash
   # Linux
   sudo ufw status
   sudo ufw allow 5000/tcp
   
   # Windows
   # 在防火墙设置中允许端口 5000
   ```

4. **查看服务器日志**
   ```bash
   tail -f logs/web_admin.log
   tail -f logs/web_admin_error.log
   ```

#### Q20: 页面显示"网络错误"或"连接失败"？

**A**: 可能的原因和解决方法：

**原因 1：后端服务器未启动**
```bash
# 启动后端服务器
python -m xagent.web_admin.server
```

**原因 2：CORS 配置错误**
在 `.env` 文件中设置：
```bash
ENABLE_CORS=true
CORS_ORIGINS=http://localhost:5173,http://localhost:5000
```

**原因 3：API 地址配置错误**
检查前端配置文件 `frontend/src/api/client.js`：
```javascript
const API_BASE_URL = 'http://localhost:5000'
```

#### Q21: 配置保存失败怎么办？

**A**: 可能的原因：

1. **配置值验证失败**
   - 检查错误提示
   - 确保 provider、layer 等字段值正确

2. **配置文件权限问题**
   ```bash
   # 检查权限
   ls -la data/session_configs.json
   
   # 修复权限
   chmod 644 data/session_configs.json
   ```

3. **磁盘空间不足**
   ```bash
   # 检查磁盘空间
   df -h
   ```

4. **文件被锁定**
   - 确保没有其他进程正在写入配置文件
   - 重启服务器

#### Q22: 前端页面显示异常或样式错乱？

**A**: 可能的原因：

1. **浏览器缓存问题**
   - 按 Ctrl+F5 强制刷新
   - 清除浏览器缓存

2. **前端构建问题**
   ```bash
   cd frontend
   rm -rf dist node_modules
   npm install
   npm run build
   ```

3. **浏览器兼容性**
   - 使用现代浏览器（Chrome、Firefox、Safari、Edge）
   - 更新浏览器到最新版本

#### Q23: 如何查看详细的错误日志？

**A**: 日志文件位置：

```bash
# 主日志文件
tail -f logs/web_admin.log

# 错误日志
tail -f logs/web_admin_error.log

# 访问日志
tail -f logs/web_admin_access.log

# 认证日志
tail -f logs/web_admin_auth.log
```

**启用详细日志**：
在 `.env` 文件中设置：
```bash
LOG_LEVEL=DEBUG
```

然后重启服务器。

#### Q24: 如何重置整个系统？

**A**: 如果需要完全重置（谨慎操作）：

1. **备份数据**
   ```bash
   cp data/session_configs.json data/session_configs_backup.json
   ```

2. **停止服务器**
   ```bash
   # 停止 Web 服务器进程
   ```

3. **清除配置文件**
   ```bash
   # 删除配置文件（或重命名）
   mv data/session_configs.json data/session_configs_old.json
   ```

4. **重启服务器**
   ```bash
   python -m xagent.web_admin.server
   ```

5. **重新导入配置**（如果需要）
   - 登录 Web 界面
   - 使用"导入配置"功能

### 性能和优化

#### Q25: 配置列表加载很慢怎么办？

**A**: 可能的原因和优化方法：

1. **配置数量太多**
   - 使用搜索和筛选功能缩小范围
   - 考虑清理不再使用的配置

2. **网络延迟**
   - 检查网络连接
   - 考虑在服务器本地访问

3. **服务器性能**
   - 增加服务器资源（CPU、内存）
   - 使用 Gunicorn 多进程模式

#### Q26: 如何提高系统性能？

**A**: 性能优化建议：

**后端优化**：
```bash
# 使用 Gunicorn 多进程
gunicorn -w 4 -b 0.0.0.0:5000 xagent.web_admin.server:app
```

**前端优化**：
- 使用生产构建（`npm run build`）
- 启用 Gzip 压缩（Nginx 配置）
- 使用 CDN 加速静态资源

**数据库优化**：
- 定期清理不再使用的配置
- 考虑使用数据库替代 JSON 文件（高级）

### 安全相关

#### Q27: 如何确保 Web 管理界面的安全？

**A**: 安全最佳实践：

1. **使用强密码**
   ```bash
   # 生成强密码
   python -c "import secrets; print(secrets.token_urlsafe(32))"
   ```

2. **启用 HTTPS**
   - 生产环境必须使用 HTTPS
   - 使用 Let's Encrypt 免费证书

3. **限制访问 IP**
   - 在 Nginx 中配置 IP 白名单
   - 只允许内网或特定 IP 访问

4. **定期更换密钥**
   - 每 3-6 个月更换管理员密码
   - 更换 JWT 签名密钥

5. **监控访问日志**
   ```bash
   # 查看访问日志
   tail -f logs/web_admin_access.log
   
   # 查看认证日志
   tail -f logs/web_admin_auth.log
   ```

#### Q28: 密码和令牌会被记录到日志吗？

**A**: 不会。系统设计时已考虑安全性：
- 密码不会记录到日志
- JWT 令牌不会记录到日志
- 错误消息不会泄露敏感信息

#### Q29: 如何防止暴力破解？

**A**: 当前版本的防护措施：
- JWT 令牌有过期时间
- 错误消息不泄露具体信息

**建议的额外措施**：
- 使用 Nginx 限制请求频率
- 配置防火墙规则
- 使用 VPN 或内网访问
- 监控认证失败日志
